# 01-02-PLAN.md

## Phase

- Phase: 1 — Planning Baseline & Codebase Mapping
- Group: B — Traceability + Phase Boundaries
- Focus: Convert roadmap and requirements into clear phase ownership boundaries using the completed codebase map.

## Goal

Connect the planning baseline to concrete execution scoping so later phases can be planned without scope drift.

This plan must:

- trace Phase 1 requirements to real planning outputs
- define what each near-term phase owns
- define what each near-term phase must not absorb
- close the boundary between Phase 1 and implementation-heavy phases

## Why This Matters

The codebase map is only useful if it helps future planners decide:

- where to work
- where not to work
- what risks belong to each phase
- how roadmap intent maps onto actual code surfaces

Without this layer, Phase 2+ would still risk overlapping ownership or drifting into adjacent subsystem work.

## Requirement Alignment

- `RQ-025` — Testable phased delivery
- `RQ-026` — Architecture consistency
- `RQ-027` — Upstream merge awareness

## Deliverables

This plan should produce:

1. A Phase 1 requirement traceability baseline.
2. Clear ownership boundaries for the next immediate phases.
3. Explicit stop conditions for Phase 1.
4. A concise view of which code areas each near-term phase should primarily target.

## Tasks

### Task B1 — Build Phase 1 requirement traceability

Map the three Phase 1 requirements to actual planning outputs:

- `RQ-025` → phased delivery, verification standards, closeout expectations
- `RQ-026` → architecture consistency, conventions, execution rules
- `RQ-027` → fork-sensitive boundaries, upstream merge awareness, divergence handling

This traceability should help future planners explain why a planning artifact exists and which requirement it satisfies.

### Task B2 — Define near-term phase boundaries

Using the roadmap plus the codebase map, define the boundaries for these near-term phases:

- Phase 2 — Layout Architecture Stabilization
- Phase 3 — Terminal Fidelity & Session UX
- Phase 4 — Remote Connection Robustness
- Phase 5 — Build & Packaging Reliability
- Phase 6 — Multi-session Terminal Hardening

For each phase, document:

- primary subsystem surface
- main risk area
- scope that belongs inside the phase
- scope that must stay outside the phase

### Task B3 — Mark Phase 1 stop conditions

Document what Phase 1 explicitly does not include now that discovery is complete.

This stop condition should prevent future planning from reclassifying Phase 1 as:

- layout design work
- terminal feature design
- remote implementation planning
- feature-level implementation decomposition

## Must-Haves

The outputs from this plan must make these true:

1. Phase 2–6 each have a distinct identity.
2. Requirement IDs are traceable to real planning outcomes.
3. Phase 1 is clearly identified as planning consolidation, not subsystem implementation planning.
4. Fork-sensitive areas are visible before later refactor phases begin.

## Out of Scope

This plan must not:

- propose implementation details for Phase 2+
- modify roadmap scope
- re-map the codebase from scratch
- create new product requirements

## Verification Criteria

This plan is complete when:

- requirement traceability is explicit
- near-term phases have clearly separated ownership boundaries
- Phase 1 stop conditions are written clearly enough to prevent scope drift
- future planning can route work to the correct phase without rediscovery
