# 01-01-PLAN.md

## Phase

- Phase: 1 — Planning Baseline & Codebase Mapping
- Group: A — Execution Standards
- Focus: Consolidate the existing planning baseline and codebase map into an execution-ready standard for later phases.

## Goal

Turn the current `.planning` baseline and `.planning/codebase/*.md` mapping outputs into a stable execution contract so future phases can plan and execute without re-discovering:

- what documents are canonical
- what conventions are mandatory
- how verification should work
- how TDD should appear in later plans
- how fork-aware work should remain reviewable and low-drift

## Why This Matters

The repository already has:

- project baseline documents
- requirement baseline
- roadmap baseline
- state baseline
- a complete 7-document codebase map

Without a formal execution standard, later phases would still need to reinterpret those inputs each time. This plan closes that gap.

## Requirement Alignment

- `RQ-025` — Testable phased delivery
- `RQ-026` — Architecture consistency
- `RQ-027` — Upstream merge awareness

## Deliverables

This plan should produce a formal execution baseline that answers the following:

1. Which `.planning` files are authoritative for later planning.
2. What each codebase document is intended to answer.
3. What future phases must inherit from Phase 1.
4. What verification standards later phases must satisfy.
5. What TDD-oriented planning language is required for behavior-changing phases.
6. What review / commit discipline should be preserved across later work.

## Tasks

### Task A1 — Define the execution baseline contract

Document the role of each planning file so later phases know:

- what to read first
- what to trust as canonical
- what to update when phase outputs change
- what not to duplicate in future planning artifacts

The contract should clearly distinguish:

- project/product context
- roadmap intent
- requirement baseline
- current codebase reality
- concerns / risk reference

### Task A2 — Convert mapping outputs into execution rules

Use the codebase map to formalize how later phases should behave when planning and executing. This should include:

- subsystem-aware scope control
- architecture consistency constraints
- evidence-driven verification expectations
- fork-sensitive change discipline
- where future phases should look before proposing changes

### Task A3 — Establish the TDD-oriented planning baseline

Define how TDD should be expressed in later plans.

For future behavior-changing phases, later plans should explicitly describe:

- expected failing behavior or missing test coverage
- minimal passing target
- refactor boundary after green state
- required verification commands
- when integration/manual checks are required beyond automated tests

## Must-Haves

The resulting standard must make these true:

1. A future phase planner can start from `.planning` without re-discovery.
2. A future executor can tell how to validate changes for the touched subsystem.
3. Phase planning uses explicit evidence, not vague project knowledge.
4. TDD language becomes operational, not aspirational.
5. Fork-awareness is built into planning discipline, not treated as an afterthought.

## Out of Scope

This plan must not drift into:

- Phase 2 layout redesign
- terminal implementation changes
- remote implementation changes
- build refactors
- test implementation for future phases
- creating new feature proposals

## Verification Criteria

This plan is complete when:

- the role of each `.planning` and `.planning/codebase` document is explicit
- later phases can inherit verification expectations without reinterpretation
- TDD-oriented planning expectations are concrete
- Phase 1 remains planning-only and does not drift into later phase design
